# Model Card: CardioRisk Cardiovascular Risk Model

Every number below is pulled directly from `backend/models/artifacts/training_metadata.json`,
`backend/logs/evaluation/best_model_metrics.json`, `backend/logs/evaluation/subgroup_metrics.json`,
and `backend/logs/evaluation/shap_feature_ranking.json` — all generated by
`backend/scripts/train.py` and `backend/scripts/feature_importance_report.py`. Nothing here is
aspirational; re-run the training pipeline and these numbers reproduce (module the exact figures
moving slightly with any change to the raw data file, hyperparameter search's random sampling
across environments, or dependency versions).

This document is written the way an ML project's methodology should be documented: not just a
metrics table, but the actual analysis — what the data looked like, what was wrong with it, what
was tried, and why each decision was made — so the numbers below can be trusted rather than taken
on faith.

## 1. Intended Use

A lifestyle/behavioral cardiovascular risk **screening** aid — not a diagnostic tool. It
estimates the probability that a person's self-reportable health profile (age, sex, BMI, blood
pressure, cholesterol/glucose category, smoking, alcohol, activity) resembles profiles associated
with cardiovascular disease in the training population. Every prediction is accompanied by a real
SHAP explanation of which factors drove the estimate up or down.

**Out of scope**: clinical diagnosis, treatment decisions, or replacing a physician's judgment.
The app surfaces this disclaimer directly in the UI (`apps/web/app/privacy/page.tsx`,
`apps/web/app/about/page.tsx`).

## 2. Data

**Primary dataset**: [Cardiovascular Disease dataset](https://www.kaggle.com/datasets/sulianova/cardiovascular-disease-dataset)
(Ulianova, 2019), CC0-licensed, 70,000 patient records, semicolon-delimited.

**Raw schema**: `id`, `age` (days), `gender` (1=female, 2=male), `height` (cm), `weight` (kg),
`ap_hi`/`ap_lo` (systolic/diastolic blood pressure), `cholesterol` (1=normal, 2=above normal,
3=well above normal), `gluc` (same scale), `smoke`, `alco`, `active` (binary lifestyle flags),
`cardio` (target: binary cardiovascular disease presence).

**Legacy dataset**: the original UCI Heart Disease dataset (`heart_disease_uci.csv`, ~920 records,
13 clinical/ECG features) is retained in `backend/data/raw/` for reference — the data loader
(`backend/src/data/data_loader.py`) auto-detects which schema a given CSV uses. The Kaggle dataset
is primary because its lifestyle features match what a self-service web questionnaire can
realistically collect (no ECG/angiography features required).

## 3. Exploratory Data Analysis

Direct analysis of the raw CSV (before any cleaning) surfaced several findings that drove every
downstream decision in this document — not assumed, measured.

**No missing values** across all 70,000 rows/13 columns.

**Class balance**: 34,979 / 35,021 (49.97% / 50.03%) — the target is effectively perfectly
balanced.

**Severe, non-random corruption in `ap_hi`/`ap_lo`**: raw values range up to 16,020 and down to
-150 (physiologically nonsensical; plausible systolic/diastolic BP is roughly 70-240 / 40-160
mmHg). Critically, **this corruption is not random noise** — it follows a systematic pattern:

| Check | Result |
|---|---|
| Rows with `ap_lo > 160` | 966 |
| ...of which plausible (40-160) after ÷10 | 926 (96%) |
| Rows with `ap_hi > 240` | 40 |
| ...of which plausible (70-240) after ÷10 | 29 (72%) |
| Rows with `ap_hi < 0` or `ap_lo < 0` | 30 |
| Rows with `ap_lo > ap_hi` (diastolic exceeding systolic) | 1,234 (1.76%) |

The ÷10 pattern is consistent with a systematic "extra trailing digit" data-entry error (e.g. a
value like `1000` really meaning `100`), not arbitrary noise — a repair heuristic is justified,
not just a defensive clip.

**This corruption matters disproportionately**, because `ap_hi`/`ap_lo` turn out to be the single
strongest predictors once cleaned:

| Feature | Correlation with target, raw | Correlation with target, clip-only | Correlation with target, repaired (this pipeline) |
|---|---|---|---|
| `ap_hi` | 0.054 | 0.418 | 0.430 |
| `ap_lo` | 0.066 | 0.296 | 0.344 |
| `age` | 0.238 | 0.238 | 0.238 |
| `cholesterol` | 0.221 | 0.221 | 0.221 |
| `weight` | 0.182 | 0.182 | 0.181 |
| `bmi` | — | 0.166 | 0.188 |
| `gluc` | 0.089 | 0.089 | 0.089 |
| gender/alco/height/smoke/active | all < 0.04 | — | — |

Simply clipping `ap_hi`/`ap_lo` to a plausible range (which the pipeline already did before this
round) unlocks most of this signal by removing the outliers' distortion of the correlation
estimate. But it still discards the *true* value for the ~950 repairable rows, replacing it with
an arbitrary boundary value — the repair approach (below) recovers real signal those rows would
otherwise lose.

**Skewness**: `ap_lo` has skewness 2.79 in the raw data (heavily right-tailed, driven by the
outliers above) and `bmi` has skewness 7.73 (also outlier-driven, compounded by the height/weight
issues below).

**Height/weight outliers**: ~53 rows have height outside a physiologically plausible adult range
(55-119cm — likely unit/entry errors) and 7 rows have weight under 30kg. These were previously
only flagged by `DataValidator`, never clipped, meaning `bmi` (and everything derived from it) was
computed from garbage for those rows.

**Duplicates**: 24 exact duplicate rows (excluding `id`).

**Multicollinearity among engineered features** (see Feature Engineering, below): once `ap_hi`/
`ap_lo` are cleaned, a mean-arterial-pressure feature tried in an earlier iteration turned out to
correlate 0.92-0.94 with the two raw columns it's derived from — essentially redundant, since MAP
is a fixed linear combination of them (`ap_lo + (ap_hi - ap_lo)/3`).

## 4. Data Cleaning

Implemented in `backend/src/data/data_loader.py::_clean_blood_pressure` and `_clean_height_weight`,
run before any feature derivation:

1. **`abs()`** on `ap_hi`/`ap_lo` — recovers a plausible value directly for the 30 negative-value
   rows.
2. **Digit repair**: for `ap_hi > 240`, use `ap_hi / 10` instead if that value falls in
   `[70, 240]`; same for `ap_lo > 160` → `ap_lo / 10` if it lands in `[40, 160]`. This is the
   primary cleaning step now, not clipping.
3. **Hard clip fallback**: `ap_hi` to `[70, 240]`, `ap_lo` to `[40, 160]` — a safety net for
   whatever the digit repair doesn't resolve (e.g. `16020` → `1602` after ÷10, still out of range
   → clipped to `240`).
4. **Drop unrepairable inconsistencies**: rows where `ap_lo > ap_hi` still holds after steps 1-3
   are physiologically impossible and dropped (not imputed) — this count falls from 1,234 to
   roughly 297 after the repair above resolves the majority automatically.
5. **Height/weight clipped before BMI derivation**: `height` to `[120, 220]`, `weight` to
   `[30, 250]` — applied *before* `bmi`/`bmi_category`/`bmi_bp_interaction` are computed, not just
   flagged post-hoc afterward.
6. **Duplicate removal**: exact duplicate rows (excluding `id`) dropped via `drop_duplicates`.

**Net effect**: 70,000 → 69,707 rows (293 dropped, 0.4%) in the current run (train/test split:
48,794 / 20,913). In exchange, `ap_hi`/`ap_lo` correlation with the target improves further (see
table above), and `ap_lo`'s skewness drops from 2.79 to a much more normal 0.33 — a small row-count
cost for a real, measured signal-quality gain on the two most important columns in the dataset.

## 5. Feature Engineering

All derived features are computed once, in `backend/src/features/derived.py::compute_derived_features`,
shared between training (`DataLoader`) and serving (`app.py::_to_feature_vector`) so the two can
never drift apart — a duplication bug (BMI computed independently in two places) that predated
this pipeline and was fixed by centralizing here.

| Feature | Formula / definition | Rationale |
|---|---|---|
| `bmi` | `weight / (height/100)²` | Standard clinical obesity measure; always computed. |
| `pulse_pressure` | `ap_hi - ap_lo` | Independent CVD risk marker distinct from either BP value alone; correlates 0.83 with `ap_hi` but only 0.23 with `ap_lo` — not redundant with either raw column. |
| `bp_category` | Ordinal 0-4, AHA thresholds on `ap_hi`/`ap_lo` (normal/elevated/stage1/stage2/crisis) | Standard clinical categorization; used as an input to `health_risk_score` and `bmi_bp_interaction` below. |
| `bmi_category` | Ordinal 0-3, WHO cutoffs (underweight/normal/overweight/obese) | Standard clinical categorization. |
| `age_bucket` | Decade bin (e.g. 3=30s, 4=40s) | Lets the Logistic Regression base model capture age nonlinearity without an explicit interaction term; also the basis for the subgroup audit below. |
| `health_risk_score` | 0-6 composite: sum of (cholesterol>1, gluc>1, smoke, alco, active==0, bp_category≥2) | Interpretable composite risk flag count; cheap for a linear model to use directly rather than learning the combination itself. |
| `bmi_bp_interaction` | `bmi × (bp_category ≥ 2)` | Captures the compounding risk of obesity and hypertension co-occurring rather than treating them as independent additive contributions. |

**Dropped**: an earlier iteration also computed mean arterial pressure
(`map_pressure = ap_lo + pulse_pressure/3`). Once `ap_hi`/`ap_lo` were properly cleaned (Section
4), `map_pressure` turned out to correlate 0.92 with `ap_hi` and 0.94 with `ap_lo` — it's a fixed
linear combination of two columns already in the model, adding almost no information beyond them
while destabilizing the Logistic Regression base model's coefficients (classic multicollinearity;
VIF-style analysis would flag this well above the common rule-of-thumb threshold of 10). It was
removed from `preprocessing.numerical_features` and `DERIVED_NUMERICAL_FEATURES` rather than kept
for its own sake.

**Not done, deliberately**: `PolynomialFeatures` / broad automatic interaction terms — with three
tree-based candidates already capturing interactions natively via splits, blanket interaction
generation adds collinear noise for marginal linear-model gain without a clear win.

Per the SHAP feature-importance diagnostic (`backend/scripts/feature_importance_report.py`,
`logs/evaluation/shap_feature_ranking.json`, computed on the served model over a 500-row test
sample), the top features by mean |SHAP value| are: `ap_hi` (0.857, dominant), `age` (0.242),
`cholesterol` category (≈0.12), `ap_lo` (0.068), `weight` (0.067), `active` (0.058),
`bmi_bp_interaction` (0.051), `bmi` (0.033), `pulse_pressure` (0.031) — confirming the derived
features earn their place rather than being dead weight, and that `pulse_pressure`'s modest but
real contribution justifies keeping it despite `map_pressure`'s removal.

## 6. Preprocessing Pipeline

Built by `backend/src/features/feature_engineering.py::FeatureEngineer`:

- **Train/test split**: stratified 70/30 (`data.test_size` in `config.yaml`), `random_state=42`.
- **Numerical features** (`age`, `height`, `weight`, `bmi`, `ap_hi`, `ap_lo`, `pulse_pressure`,
  `health_risk_score`, `bmi_bp_interaction`): median imputation (defensive — the cleaned dataset
  has no missing values, but this guards against future data sources that might) + `StandardScaler`.
- **Categorical features** (`sex`, `cholesterol`, `gluc`, `smoke`, `alco`, `active`,
  `bp_category`, `bmi_category`, `age_bucket`): most-frequent imputation + `OneHotEncoder`
  (`handle_unknown="ignore"`).
- All of this is fit on the training split only and applied to both splits — standard practice to
  avoid leaking test-set statistics into preprocessing.

## 7. Model Selection Strategy

Five candidates are evaluated: Logistic Regression, Random Forest, LightGBM, XGBoost, and a
Stacking ensemble of the first four. SVM is deliberately excluded — its only SHAP explainer
option, `KernelExplainer`, took 30-60 seconds per prediction in practice, incompatible with an
interactive API.

**Cheap screening before full tuning** (`ModelTrainer._screen_candidates`,
`training.screening` in `config.yaml`): across repeated retrains of this project, Random Forest
never won on ROC-AUC — it was consistently ~0.001-0.003 behind LightGBM/XGBoost while taking
5-10x longer to tune (its search space allows up to 600 trees per fit; 50 Optuna trials × 5-fold
CV means up to 250 full forest fits). Rather than hardcoding "give Random Forest less budget",
every candidate first runs a fast, untuned 3-fold CV pass:

| Candidate | Screening CV ROC-AUC (untuned, 3-fold) |
|---|---|
| LightGBM | 0.7996 |
| XGBoost | 0.7909 |
| Logistic Regression | 0.7954 |
| Random Forest | 0.7703 |

Candidates within `training.screening.margin` (0.01 ROC-AUC) of the best screening score keep the
full Optuna budget (50 trials); those further behind get `training.screening.reduced_trials` (10)
instead. In this run, Random Forest's screening score (0.7703) was 0.0294 behind the leader
(LightGBM, 0.7996) — well outside the 0.01 margin — so it was automatically dropped to a 10-trial
budget. This is data-driven, not a hardcoded exclusion: if a future dataset or feature change made
Random Forest competitive again, it would automatically earn the full budget back.

**Effect**: Random Forest's tuning time dropped from ~40-50 minutes (50 trials) to roughly
8-9 minutes (10 trials) in this run, while still reaching a competitive tuned CV ROC-AUC of 0.8001
— within 0.003 of the fully-tuned LightGBM/XGBoost, at a fraction of the compute cost. Total
pipeline wall-clock time dropped from over 100 minutes (previous round, full budget for every
candidate) to about 42 minutes.

**Selection metric**: CV ROC-AUC (not raw accuracy, which can favor a worse-calibrated,
less-discriminating model), with CV F1 as a tie-breaker within `training.model_selection.tie_tolerance`
(0.002).

## 8. Training

**Hyperparameter tuning** (`backend/src/training/tuning.py::HyperparameterTuner`): Optuna,
TPE sampler, per-model search spaces (e.g. LightGBM tunes `n_estimators`, `num_leaves`,
`learning_rate`, `min_child_samples`, `reg_alpha`, `reg_lambda`). Each trial is scored via
`StratifiedKFold` CV on the training split only — this is intentionally single-level CV, not
nested CV: the held-out test set is never touched during tuning, so it still yields an honest
generalization estimate without paying for a full nested-CV loop (which would cost roughly 5x
more compute at this dataset size for no meaningful benefit).

**5-fold stratified CV results** (tuned hyperparameters shown for the top candidates):

| Model | CV ROC-AUC | CV F1 | CV Accuracy | Tuned hyperparameters |
|---|---|---|---|---|
| Logistic Regression | 0.7956 | 0.7114 | 0.7307 | C=0.0261 |
| Random Forest (reduced budget) | 0.8001 | 0.7191 | 0.7342 | n_estimators=273, max_depth=9, min_samples_leaf=11 |
| **LightGBM (selected)** | **0.8027** | **0.7255** | **0.7359** | n_estimators=247, num_leaves=15, learning_rate=0.0353, min_child_samples=39 |
| XGBoost | 0.8026 | 0.7242 | 0.7354 | n_estimators=536, max_depth=4, learning_rate=0.0179, subsample=0.77 |
| Stacking (single-split validation) | 0.8003 | 0.7246 | 0.7341 | meta-learner (Logistic Regression) over the four models above |

LightGBM is selected: XGBoost's raw ROC-AUC (0.8026) is within the 0.002 tie-tolerance of
LightGBM's (0.8027), so the F1 tie-breaker applies — LightGBM's F1 (0.7255) beats XGBoost's
(0.7242), so LightGBM wins. The Stacking ensemble (evaluated on a single stratified
train/validation split rather than a full outer k-fold — see the `_train_stacking` docstring for
why, in short: a `StackingClassifier` already runs its own internal CV to build leakage-free
meta-features, so an outer k-fold would roughly square the cost for no needed variance estimate)
scored 0.8003, below LightGBM's margin, so it did not win this run. This is an honest run-to-run
outcome, not cherry-picked — a previous round's Stacking ensemble did win by a similar margin in
the other direction; single-split evaluation carries some inherent variance, which is disclosed
here rather than hidden.

**Calibration**: the selected model is wrapped in `sklearn.calibration.CalibratedClassifierCV`
(isotonic regression, 5-fold) so its output probabilities are clinically meaningful, not just
well-ranked. Isotonic was chosen over Platt/sigmoid scaling because it makes fewer assumptions
about the shape of the miscalibration and reliably matches-or-beats Platt scaling once the
calibration set exceeds roughly 1,000 samples (this pipeline's training split has 48,794 rows) —
consistent with a 2025 clinical-ML study on post-hoc calibration for heart disease prediction.

## 10. Fairness / Subgroup Audit

Aggregate metrics can mask meaningful gaps in specific subgroups. Breakdown by sex and age bucket
(`ModelEvaluator.evaluate_by_subgroup`, `logs/evaluation/subgroup_metrics.json` — reporting only,
does not change training or selection):

| Sex | n | Accuracy | ROC-AUC | Brier |
|---|---|---|---|---|
| Female (0) | 13,713 | 0.7392 | 0.8046 | 0.1790 |
| Male (1) | 7,200 | 0.7388 | 0.8030 | 0.1792 |

Sex-based performance is close (within 0.002 ROC-AUC) — no material fairness concern there.

| Age bucket (decade) | n | Accuracy | ROC-AUC | Brier |
|---|---|---|---|---|
| 30s | 537 | 0.8492 | 0.8343 | 0.1159 |
| 40s | 5,996 | 0.7917 | 0.8222 | 0.1562 |
| 50s | 10,585 | 0.7146 | 0.7721 | 0.1921 |
| 60s | 3,794 | 0.7082 | 0.7270 | 0.1880 |

**A real, disclosed gap**: the model discriminates and calibrates noticeably worse for patients in
their 50s-60s (ROC-AUC 0.73-0.77, Brier 0.19) than for patients in their 30s-40s (ROC-AUC 0.82-0.83,
Brier 0.12-0.16) — and the 50s-60s buckets make up the majority (69%) of the test set. This likely
reflects that cardiovascular risk becomes harder to discriminate from lifestyle/vitals alone as
more of an older population develops disease regardless of individual risk-factor variation (a
ceiling effect), rather than a data or modeling bug. Flagged here rather than smoothed over.

## 11. Explainability

Every `/predict` response includes real signed SHAP contributions
(`backend/src/evaluation/explainer.py`), not a static feature-importance list:

- **TreeExplainer** for tree-ensemble candidates (Random Forest, LightGBM — **the currently served
  model** —, XGBoost): fast, exact.
- **LinearExplainer** for Logistic Regression.
- **WeightedContributionExplainer**, used only when the served model is the Stacking ensemble
  (not the case in the current run, but implemented and tested): combines each base estimator's
  own fast/exact SHAP contributions, weighted by the meta-learner's fitted coefficients. This is
  an explicitly disclosed *approximation*, not a Shapley-exact value for the full stack (it
  ignores any nonlinearity a non-linear meta-learner could in principle capture — moot here since
  the meta-learner is Logistic Regression). Chosen over `shap.KernelExplainer` on the meta-learner,
  which would cost the same 30-60s/prediction that already ruled out SVM. When Stacking is served,
  `/predict` latency rises to ~3-5s (still far below Kernel's 30-60s) since it runs four
  sub-explainers per request; with LightGBM served (the current state), latency is ~0.18s.

The served model is a `CalibratedClassifierCV` wrapping the winning candidate; the explainer
dispatch unwraps this automatically (`backend/src/utils/model_introspection.py::unwrap_calibrated`)
before selecting an explainer type, so calibration never silently degrades explanation
speed/quality.

A `baseline_probability` (the SHAP expected value — the model's average prediction over a
background sample) is also returned, so the UI can frame each result as "started at X%, your
factors moved it to Y%."

## 12. Why not SMOTE / class rebalancing

The dataset is close to perfectly balanced (34,979 / 35,021 positive in the source data). SMOTE
and other imbalanced-learn resampling techniques are designed for meaningfully skewed classes;
applying them here would synthesize implausible patients (interpolating between real BP/BMI/age
combinations) for no benefit, and would require switching to an `imblearn.pipeline.Pipeline` to
avoid leakage across CV folds. `class_weight="balanced"` / `"balanced_subsample"` on the
tree-based candidates is sufficient given the near-perfect balance.

## 13. Why not deep learning

With ~18 tabular features (9 raw + 9 derived/categorical, after the `map_pressure` pruning) over
~70,000 rows, this is squarely gradient-boosted-tree/linear territory. Recent tabular-deep-learning
benchmarks (2024-2025) show methods like FT-Transformer and TabR are now competitive with tuned
GBDTs on *some* datasets, but tree-based models still win or tie on most small-to-medium tabular
problems, especially under a hyperparameter-tuning budget like this one's. A neural net would also
forfeit the fast, exact `TreeExplainer` SHAP path this system depends on for sub-second `/predict`
latency — not worth it for this dataset size and shape.

## Limitations

- **Not diagnostic.** No ECG, imaging, lab-confirmed lipid panel, or clinical history — only
  self-reportable lifestyle and vitals.
- **Self-reported data.** The training data's smoking/alcohol/activity fields are self-reported in
  the source survey and subject to reporting bias.
- **Subgroup performance gap by age.** See Section 10 — ROC-AUC/calibration are noticeably worse
  for patients in their 50s-60s than in their 30s-40s. Sex-based performance is comparable. Any
  real-world deployment should account for this age-related discrimination gap rather than
  presenting a single aggregate accuracy figure.
- **Stacking ensemble's SHAP explanations, when served, are an approximation.** See Section 11 —
  `WeightedContributionExplainer` is not Shapley-exact for the full stack, and its `/predict`
  latency (~3-5s) is higher than a single tree model's (~0.18s) as a result of this design choice.
- **Cholesterol/glucose are categorical, not lab values.** The model consumes a 3-level category
  (normal/above/well above), not a continuous mg/dL reading, because that's what the training
  dataset provides.
- **Document-upload extraction (RAG pipeline) is not independently validated.** The PDF/DOCX
  auto-fill feature (`backend/src/rag/pipeline.py`) is an LLM-based convenience for form-filling,
  not a clinically validated extraction tool — the UI requires the user to review and confirm
  every extracted value before it's used for a prediction.

## Ethical Considerations

This is an educational/portfolio project, not a certified medical device, and is presented as such
throughout the app (see the persistent disclaimer in `apps/web/app/components/SiteShell.tsx`'s
footer). Given the dataset's self-reported nature and the disclosed age-related subgroup
performance gap above, it should not be used to make or influence real clinical decisions.
